# 本地 GPU 批处理 OCR-RAG 流水线设计

生成日期：2026-07-20

作者：张春

说明：AI 根据当前 `OCR-Harness / Lumia ScriptorRAG` 项目代码、历史对话、已有部署文档与本地目录结构整理生成。

## 1. 背景

用户需要对 `C:\Users\35160\Desktop\中医古今书籍10200册\中医古今书籍10200册` 进行一次本地化大批量处理，目标不是调用远程 OCR API，而是利用本机显卡完成：

- 目录扫描与任务清单生成
- 复用已有 `.md` / `.txt` 结果
- 其余 PDF 走本地 GPU OCR
- OCR 结果进行 Markdown 排版清洗
- 清洗后增量写入 RAG
- 前端实时展示处理进度、质量和命中情况
- 后续支撑容量统计、召回测试和可审计报告

## 2. 目标

### 2.1 业务目标

- 10200 册目录可以断点续跑。
- 优先复用已有 `.md` / `.txt`，减少重复 OCR。
- 其余 PDF 必须走本地 GPU 加速。
- 结果要可追溯到原文件、处理引擎、质量分和输出路径。
- 每个批次都能直接进入 RAG，支持边跑边查。

### 2.2 技术目标

- 保持当前项目的 FastAPI、前端和 RAG 结构不重写。
- 将批处理任务与在线 API 解耦。
- 把“目录扫描、复用判断、OCR、排版、RAG、统计”拆成清晰边界。
- 让批处理过程能被前端实时查看，而不是只靠日志。

## 3. 非目标

- 不把远程 OCR API 作为主链路。
- 不先做模型训练或端到端大模型微调。
- 不强制把所有文档一次性重写成同一种格式。
- 不把项目改造成单纯的云端 SaaS。

## 4. 硬件与约束

- 目标机器显卡：`RTX 5070 Ti`。
- 该任务应该按“本地 GPU 优先、CPU 负责控制流”的方式设计。
- 批处理必须支持长时间运行、断点恢复和失败重试。
- 如果单个模型过重，必须允许降级到更轻的 OCR 引擎。

## 5. 方案选择

### 5.1 总体方案

采用“三层本地流水线”：

1. 扫描层：遍历目录，生成任务清单，识别可复用结果。
2. 处理层：对未复用 PDF 使用本地 GPU OCR，并进行 Markdown 排版。
3. 索引层：将结果增量写入 RAG，并记录统计指标。

### 5.2 复用规则

- 若 PDF 同名或高相似度对应的 `.md` / `.txt` 已存在，先做质量评估。
- 若质量评估达到阈值，则直接复用，不重复 OCR。
- 若质量不足或没有基线文件，则进入 OCR 队列。
- 复用结果与 OCR 结果都要写入统一日志。

### 5.3 OCR 引擎优先级

默认建议以下策略：

- 主引擎：百度系最新文档 OCR 模型优先，例如 `PaddleOCR-VL`。
- 轻量回退：`PP-OCRv6` 或当前项目可用的 `PaddleOCR` 适配器。
- 结构化文档回退：`Surya`、`Docling`、`MinerU`。
- 实验引擎：`Unlimited-OCR` 仅作为长文档或特定样本的对照测试。

说明：
- `PaddleOCR-VL` 更适合作为文档解析主引擎。
- `Unlimited-OCR` 更适合长序列、大页数连续文本的实验场景。
- 当前设计保留引擎插件化，不把单一模型写死。

## 6. 架构

```text
目录扫描器
  -> 复用判定器
  -> OCR GPU Worker
  -> Markdown 排版清洗器
  -> 增量 RAG 写入器
  -> 统计聚合器
  -> 前端监控页
```

### 6.1 目录扫描器

职责：

- 扫描目标目录下所有 PDF。
- 同步识别目录中已有 `.md` / `.txt` 基线文件。
- 为每个 PDF 生成标准任务记录。
- 记录原始路径、相对路径、文件大小、修改时间和复用候选。

输出：

- 批次 manifest。
- 每个任务的状态记录。
- 预估处理量和复用量。

### 6.2 复用判定器

职责：

- 对 PDF 文件名与已有 `.md` / `.txt` 文件名做模糊匹配。
- 对命中内容做质量评分。
- 判断是否直接复用结果。

规则：

- 高质量复用优先。
- 复用结果低于阈值时进入 OCR。
- 复用判断必须可解释，保存命中来源与评分。

### 6.3 OCR GPU Worker

职责：

- 读取任务清单。
- 使用本地显卡执行 OCR。
- 产出 Markdown 与引擎元数据。
- 记录每一步进度。

要求：

- 不使用远程 API。
- 不把 `CUDA_VISIBLE_DEVICES=""` 写死为禁用。
- 允许从环境变量或启动脚本中显式指定 GPU。

### 6.4 Markdown 排版清洗器

职责：

- 清理 OCR 噪声。
- 修正标题、列表、段落、表格和标点。
- 保留原文结构与可追溯性。

输出：

- 标准化 Markdown。
- 排版质量统计。

### 6.5 增量 RAG 写入器

职责：

- 将复用内容和 OCR 内容都写入本地索引。
- 支持批次内增量更新。
- 允许后续重建或局部重建。

### 6.6 统计聚合器

职责：

- 记录总文件数、已复用数、已 OCR 数、失败数、平均质量分、引擎分布。
- 记录处理耗时、每批吞吐量、文件大小分布。
- 为后续“AI 鉴定报告”和知识库容量展示提供原始数据。

### 6.7 前端监控页

职责：

- 实时显示批次总进度。
- 显示当前处理文件、当前引擎、质量分、复用状态。
- 允许查看单个文件的原文、Markdown、结果路径和检索命中。

## 8. 处理流程

### 8.1 第一阶段：扫描与分组

1. 扫描全部 PDF。
2. 收集已有 `.md` / `.txt` 结果。
3. 生成批次 manifest。
4. 按目录树或批次大小分组，避免一次性任务过大。

### 8.2 第二阶段：基线复用

1. 为每个 PDF 匹配候选 `.md` / `.txt`。
2. 计算基线质量。
3. 若达标则直接写入索引并标记为复用。
4. 若不达标则进入 OCR 队列。

### 8.3 第三阶段：GPU OCR

1. 从高优先级 OCR 引擎开始尝试。
2. 失败时自动降级到下一引擎。
3. 保存每个引擎的成功/失败与质量分。
4. 生成 Markdown 原始结果。

### 8.4 第四阶段：排版清洗

1. 将 OCR 结果送入 Markdown 清洗器。
2. 修正标题层级、列表、段落与表格。
3. 删除无意义符号、重复空白和乱码片段。
4. 输出可读 Markdown。

### 8.5 第五阶段：RAG 增量入库

1. 将复用结果和 OCR 结果统一写入 RAG。
2. 记录来源路径与批次 ID。
3. 增量刷新索引状态。
4. 支持后续检索验证和召回测试。

## 11. 验证方式

### 11.1 基础验证

- 目录扫描数是否等于实际 PDF 数。
- 复用文件是否正确进入索引。
- OCR 文件是否生成 Markdown。
- 清洗前后内容是否可读。
- RAG 是否能检索到本批次内容。

### 11.2 质量验证

- 质量分是否达到阈值。
- 召回结果是否来源于不同文档。
- 查询时是否能返回可追溯出处。
- 是否能导出报告并回看过程。

### 11.3 性能验证

- 单卡是否能持续工作。
- GPU 利用率是否稳定。
- 批处理吞吐量是否可接受。
- 是否支持中断后续跑。

## 12. 风险与对策

- 风险：10200 册体量过大。  
  对策：分批、断点续跑、增量入库。

- 风险：GPU 显存不足。  
  对策：优先高质量模型，必要时自动降级。

- 风险：复用规则误判。  
  对策：保留命中来源与质量分，可人工复核。

- 风险：长时间批处理出错。  
  对策：文件级状态持久化和日志重放。
