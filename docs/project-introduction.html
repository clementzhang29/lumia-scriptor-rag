项目定位

Lumia ScriptorRAG 不是单纯的 RAG 聊天页，也不是单引擎 OCR 工具，而是一条围绕“文档从不可用到可阅读、可检索、可引用”的完整流水线。OCR 智能体是入口，排版清洗是中间层，RAG 是下游知识应用。

光棱 OCR

多 OCR 引擎路由、质量评分、失败回退。

2

字炉排版

清洗 Markdown、修复结构、去除噪声。

3

文枢 RAG

多格式建库、来源去重、原典优先。

4

星轨模型

OpenAI 兼容与多厂商 LLM Provider。

5

WebUI / API

前端交互、结果导出、二次开发入口。

PDF / 扫描件 / 本地资料
  → 光棱 OCR：Surya / MinerU / Docling / Marker / PaddleOCR / Nougat

实现过程

一、跑通 OCR

先验证 Surya OCR 端到端工作流，再补充 MinerU、Docling 等引擎适配和依赖处理。

二、补齐排版

将 OCR 输出整理成 Markdown，并通过清洗器去除噪声字符、修复段落和标题结构。

三、融合 RAG

把清洗文档导入索引，要求检索结果尽量来自不同文档，古籍原典优先。

四、产品化前端

加入 OCR、导览、排版、RAG、模型、帮助等入口，顶部导航最终精简为图标 + 短文案。

五、稳定本地运行

处理 Windows Python 启动、端口占用、Vue History、FastAPI 静态资源白屏等问题。

遇到的问题与解决

问题解决方案
Windows 无法识别 python改用 py -3 启动，并在脚本中自动探测可用 Python。
MinerU / torch GPU DLL 依赖冲突补充 doclayout_yolo、ultralytics，必要时设置 CUDA_VISIBLE_DEVICES="" 降级 CPU。
前端打开后白屏挂载 /assets 静态资源，避免 JS 被 SPA fallback 返回成 HTML。
导航文字竖排越界改为 OCR、导览、排版、RAG、模型、帮助等短标签，并使用单行胶囊式导航。
RAG 检索来源过窄提高不同文档来源覆盖率，并提升古籍原典权重。
API Key 安全不提交密钥，Provider 配置后续应加密落盘。

简历用介绍

Lumia ScriptorRAG（序光文枢 RAG）是一个多模型智能编排的本地文档数字化与知识库系统，围绕“光棱 OCR → 字炉排版 → 文枢 RAG”的完整链路设计。项目集成 Surya、MinerU、Docling、Marker、PaddleOCR、Nougat 等 OCR 引擎，支持文档类型分析、自动路由、质量评分、失败回退、LLM 增强校正和 Markdown 导出；同时扩展通用 RAG 能力，支持多格式建库、来源去重、原典权重提升和可追溯问答。本人负责整体架构设计、FastAPI 后端接口、OCR Pipeline 编排、LLM Provider 抽象、多格式文档导入、BM25 检索模块、Vue3 前端产品化、本地启动脚本与交付文档建设。
